shot-scraper 1.11 提升服务器启动可靠性与命令行参数一致性

新闻时间: 2026-07-13T04:43:02.962448+00:00 · 来源: rss

西蒙·威利森（Simon Willison）发布了 shot-scraper 1.11 版本，这是一次以实用性为导向的维护性更新，旨在增强该工具在真实自动化工作流中的稳健性。本次发布聚焦于两项关键改进：大幅延长多页截图与视频录制过程中本地服务器启动的等待超时时间，并统一各子命令的 JavaScript 注入方式。这些改动体现了这款命令行工具正从功能探索阶段转向注重可靠性与开发者体验的成熟期。

最具实际价值的改进是解决了用户在 server 模式下长期面临的一个痛点——尤其是在执行 multi 和 video 命令时。此前，工具仅等待 1 秒钟便判定本地启动的 Web 服务器未能响应并直接失败。而在实际场景中（例如在较慢的机器上运行，或服务内容复杂的静态站点），这一硬编码延迟常导致自动化流程无故中断。1.11 版本将其替换为自适应轮询机制，最多等待 30 秒，并持续检测端口是否就绪，直至服务器成功响应或超时终止。此举显著提升了 CI 环境或资源受限设备中截图与视频生成任务的稳定性。

同样重要的是，--js-file 参数现已扩展至五个核心命令（shot-scraper、pdf、html、accessibility 和 har）。用户不再需要将 JavaScript 逻辑直接写入 --javascript 参数中，而是可通过外部文件路径、标准输入（用 - 表示）或 GitHub 托管脚本（如 gh:username/script）来加载脚本。这更利于版本控制、代码复用和复杂浏览器端逻辑的可读性。multi 命令也同步支持 YAML 配置中的 js_file: 键，使多步骤捕获流程可完全声明式定义。

其他细节优化还包括为 shot-scraper javascript 和 html 命令新增 --timeout 参数，使其与其他子命令保持接口一致。尽管这些更新均未引入颠覆性功能，但它们协同作用，体现了对降低使用门槛、规避常见失败、统一命令接口的持续投入。对于文档维护者、质量保障工程师以及构建自动化视觉回归测试或演示视频流水线的开发者而言，shot-scraper 1.11 带来了静默却切实的可靠性提升。
