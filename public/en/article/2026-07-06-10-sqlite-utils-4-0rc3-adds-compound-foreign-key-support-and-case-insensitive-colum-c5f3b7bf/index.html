sqlite-utils 4.0rc3 adds compound foreign key support and case-insensitive column handling

News time: 2026-07-06T18:31:54.093488+00:00 · Source: rss

Simon Willison has released sqlite-utils 4.0rc3 — the third release candidate for the upcoming major version of his popular Python library for interacting with SQLite databases via CLI and Python API. Originally aiming for a stable 4.0 release over the weekend, Willison instead found the changelog expanding significantly as he triaged issues and pull requests using advanced large language models — specifically Claude Fable 5 and GPT-5.5 — underscoring how AI-assisted development is increasingly shaping open-source maintenance workflows.

The standout feature in rc3 is full support for compound (or composite) foreign keys: multi-column foreign key constraints that reference multiple columns in a parent table. This capability aligns sqlite-utils more closely with SQLite’s native foreign key syntax and enables robust relational modeling for complex schemas — such as many-to-many junction tables where two or more columns jointly form a foreign key relationship. However, implementing this required a subtle but breaking change to the table.foreign_keys property in the Python API, which now returns richer metadata including column-name mappings and constraint details previously omitted.

Equally impactful is the library’s new adherence to SQLite’s official case-insensitive column name resolution behavior. Previously, sqlite-utils treated column names with strict case sensitivity in certain contexts — a deviation from SQLite’s documented semantics. The update harmonizes behavior across CLI commands (e.g., sqlite-utils add-foreign-key) and Python methods (e.g., Table.add_foreign_key()), ensuring consistent column matching whether users specify 'Author_ID', 'author_id', or 'AUTHOR_ID'. This change ripples through internal logic, affecting schema introspection, column lookup, and transform operations.

These improvements reflect sqlite-utils’ evolution from a lightweight utility into a production-grade tool for data engineers and developers who treat SQLite not just as an embedded database but as a first-class data platform — especially when paired with tools like Datasette. While rc3 delays the final 4.0 release, it ensures that foundational behaviors are correct before stabilization. Developers upgrading from v3.x should audit their code for assumptions about table.foreign_keys structure and column name casing — both of which now behave differently, intentionally and consistently.
