sqlite-utils 4.0rc3 Adds Compound Foreign Keys and Case-Insensitive Column Handling

News time: 2026-07-06T05:40:08+00:00 · Source: rss

Simon Willison has released sqlite-utils 4.0rc3 — the third release candidate for the upcoming major version of his popular Python library for programmatic SQLite interaction. Originally aiming for a stable 4.0 release over the weekend, Willison found the changelog expanding significantly as he triaged issues and reviewed pull requests with assistance from AI models including Claude Fable 5 and GPT-5.5. This reflects both the growing complexity of the project and an evolving development workflow where large language models help surface edge cases and documentation gaps.

The standout feature in rc3 is full support for compound (or composite) foreign keys — multi-column foreign key constraints that reference multiple columns in a parent table. SQLite has supported this syntax since long before 2026, but sqlite-utils previously lacked introspection and declarative creation capabilities for such constraints. Now developers can inspect existing compound foreign keys via table.foreign_keys and define new ones programmatically — a capability especially valuable for data migration, schema validation, and building robust CLI-driven database tooling.

This enhancement required a subtle but breaking change to the table.foreign_keys API: the returned objects now include additional fields like columns, other_table, and other_columns, enabling accurate representation of multi-column relationships. Simultaneously, sqlite-utils now aligns with SQLite’s native behavior by treating column names case-insensitively — meaning 'id', 'ID', and 'Id' are treated as equivalent when referencing or matching columns. This change ripples across several internal subsystems, including column lookup, foreign key resolution, and table alteration logic.

For Python developers who rely on sqlite-utils for scripting, ETL pipelines, or lightweight ORM alternatives, these updates improve fidelity with SQLite’s actual semantics while enabling more sophisticated relational modeling. Though still pre-release, rc3 signals that the stable 4.0 version will prioritize correctness, standards compliance, and extensibility over backward compatibility — a trade-off justified by the library’s maturing role in the Python data ecosystem.
