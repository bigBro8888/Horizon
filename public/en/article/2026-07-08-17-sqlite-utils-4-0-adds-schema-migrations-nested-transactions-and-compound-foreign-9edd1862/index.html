sqlite-utils 4.0 Adds Schema Migrations, Nested Transactions, and Compound Foreign Keys

News time: 2026-07-08T04:29:37.384585+00:00 · Source: rss

Simon Willison has released sqlite-utils 4.0 — the first major version of this widely adopted Python library and CLI tool for SQLite in over five years. As the 124th release overall and the successor to v3.0 (November 2020), this update delivers three foundational enhancements: built-in database schema migrations, nested transaction support via db.atomic(), and native handling of compound foreign keys. These features collectively address long-standing gaps in lightweight SQLite tooling — especially for developers building data-centric applications, prototypes, or personal analytics tools where robust yet simple schema evolution is essential.

The new migration system lets users define incremental, versioned changes to their database structure using plain Python functions. Each migration receives a Database instance and can perform arbitrary operations — creating tables, adding columns, transforming schemas with table.transform(), or inserting seed data. Crucially, table.transform() implements SQLite’s recommended pattern for non-trivial schema changes: it creates a new table with updated definitions, copies data safely, then swaps names — bypassing SQLite’s limited ALTER TABLE capabilities. Migrations are tracked automatically in a special _migrations table, ensuring idempotent execution and preventing duplicate application.

Running migrations is straightforward: uvx sqlite-utils migrate data.db migrations.py applies all pending functions in order. Developers can also list migrations (--list), run verbosely (--verbose) to see before/after schema diffs, or stop partway (--stop-before). The implementation draws from Willison’s earlier sqlite-migrate project — now matured and officially integrated — reflecting real-world validation over multiple years. Meanwhile, db.atomic() enables true nested transactions, letting inner atomic() blocks roll back independently without affecting outer ones — a significant improvement over SQLite’s default flat transaction model.

Compound foreign key support aligns sqlite-utils more closely with SQLite’s full SQL standard: developers can now declare foreign keys referencing multiple columns (e.g., (order_id, item_seq) → (orders.id, order_items.seq)), enabling accurate modeling of many-to-many relationships and composite primary keys. While foreign key enforcement remains disabled by default in SQLite (requiring PRAGMA foreign_keys = ON), sqlite-utils now provides ergonomic Python and CLI interfaces to define and manage such constraints correctly. Together, these features make sqlite-utils 4.0 not just an upgrade — but a step toward treating SQLite as a production-ready, evolvable database layer.
