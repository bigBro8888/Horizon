DOOMQL: A Doom-like Game Engine Entirely in SQLite SQL

News time: 2026-07-14T04:15:20.855259+00:00 · Source: rss

DOOMQL is not just another terminal game—it’s a radical reimagining of what a database can do. Built by Peter Gostev using experimental tooling (including GPT-5.6 Sol), this project transforms SQLite from a passive data store into the active runtime engine for a fully playable, first-person retro shooter. Every frame of gameplay—movement, collision detection, enemy AI, weapon firing, health tracking, and even pixel-by-pixel screen rendering—is computed not by C or Python logic, but by carefully orchestrated SQL queries executed inside SQLite. The result is a mesmerizing text-mode Doom clone rendered in ASCII-style pixel art, running entirely within a single SQLite database file.

At its core lies a recursive Common Table Expression (CTE) that implements a full raycasting renderer—a technique historically reserved for compiled languages like C. This SQL-based ray tracer calculates wall distances, applies perspective projection, handles texture mapping via lookup tables, and composites sprites—all without external libraries or procedural code. The game loop runs at ~30 FPS via a lightweight Python host script that polls keyboard input and triggers SQL execution; the database itself holds all mutable state: player position, ammo count, enemy locations, and even the current framebuffer stored as rows in a frame_pixels view with (x, y, r, g, b) columns.

What makes DOOMQL especially compelling is its introspectability and composability. Because all state lives in SQLite, developers can inspect, query, and visualize it in real time using tools like Datasette. With the new Datasette Apps plugin, users can build live web dashboards—like an auto-refreshing 3D viewport or a tactical top-down minimap—entirely through declarative HTML/JS apps that issue SQL directly against the same database powering the game. One user even generated both the viewport and minimap using natural-language prompts to Claude (Fable 5), demonstrating how tightly SQL, UI, and game logic are now unified.

DOOMQL doesn’t aim for production performance or scalability—it’s a conceptual proof-of-concept that pushes SQL beyond its traditional boundaries. Yet its implications ripple across systems programming, education, and tooling design: it challenges assumptions about where computation belongs, invites new ways to debug and observe complex state machines, and proves that expressive declarative logic can drive rich interactive experiences—even in a terminal.
