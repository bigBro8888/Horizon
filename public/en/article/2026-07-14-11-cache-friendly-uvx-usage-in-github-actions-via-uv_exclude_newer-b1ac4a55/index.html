Cache-Friendly uvx Usage in GitHub Actions via UV_EXCLUDE_NEWER

News time: 2026-07-14T04:15:20.855210+00:00 · Source: rss

A practical optimization for Python-based CI/CD pipelines has emerged: using uvx—the tool execution subcommand of the ultra-fast Rust-based Python package installer uv—in a way that avoids redundant PyPI downloads across GitHub Actions workflow runs. Simon Willison, a well-known developer and technical writer, recently shared a refined pattern that makes uvx both reproducible and cache-efficient by leveraging the UV_EXCLUDE_NEWER environment variable.

The core idea is simple but powerful: pin the effective 'time horizon' for package resolution by setting UV_EXCLUDE_NEWER to a fixed date (e.g., "2026-07-12"). This tells uv to ignore any package versions published after that date—even if newer versions exist on PyPI. As a result, uvx black, uvx ruff, or any other tool invocation will consistently resolve to the latest version available as of that cutoff, enabling deterministic behavior without sacrificing freshness.

Crucially, this date is then embedded directly into the GitHub Actions cache key—such as cache-key: uvx-${{ env.UV_EXCLUDE_NEWER }}. Because the cache key changes only when the date is manually updated, repeated workflow runs reuse the same cached tool installation (including wheels and dependencies), skipping network fetches entirely. Willison reports measurable time savings—often over 40 seconds per job—by eliminating repeated downloads and wheel builds from PyPI.

This approach also sidesteps common pitfalls of naïve caching: unlike caching based solely on tool names or versions, it avoids cache invalidation storms caused by frequent upstream releases. It’s more flexible than hardcoding exact versions (which require constant maintenance) and more reliable than relying on uv’s default behavior—which, by design, always checks for the newest available version unless constrained. An open issue in the astral-sh/setup-uv repository even confirms community demand for making such caching the default behavior in official setup actions.
