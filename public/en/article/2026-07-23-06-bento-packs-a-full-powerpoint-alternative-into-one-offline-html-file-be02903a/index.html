Bento packs a full PowerPoint alternative into one offline HTML file

News time: 2026-07-23T04:30:17.189283+00:00 · Source: hackernews

Bento is a new kind of presentation tool that fits entirely inside a single HTML file — no servers, no cloud accounts, no installation required. Open it in any modern browser and you can edit slides, run animations, present full-screen, print to PDF, and even collaborate live with others. The entire application — editor, renderer, collaboration logic, and default content — is embedded in that one file, which clocks in at about 560 KB and works completely offline after initial load.

The project emerged from developer frustration with AI-assisted slide creation: while tools like Claude Code could generate beautiful decks from prompts, making small edits meant diving into raw code or re-running agents. Bento closes that loop by offering a visual editor atop a clean, human-readable JSON data section near the top of the file — something developers can grep, diff, or feed directly to LLMs for transformation. It builds on reveal.js but adds custom layers for local-first sync and lightweight compression using base64 + DecompressionStream in the browser.

Collaboration happens via an encrypted blind relay: when users share a deck, they exchange a link that connects them through a minimal relay server. That server only forwards encrypted payloads — it never sees slide content, user identity, or edit history. This design supports real-time co-editing without compromising privacy or requiring backend infrastructure. You can try it immediately via the guestbook demo, where multiple people simultaneously edit a shared page without signing in.

Bento is fully open source (MIT licensed), with all code on GitHub. Its architecture invites extension: drop a .pptx into ChatGPT or Claude, ask it to convert to Bento’s JSON format, and paste the result into the file. Or host decks on GitHub Pages, email them as attachments, or AirDrop them between devices — all while retaining full editing capability. As one commenter noted, this reflects a broader shift toward local-first, agent-friendly tools that treat the browser not as a thin client, but as a self-sufficient runtime.
