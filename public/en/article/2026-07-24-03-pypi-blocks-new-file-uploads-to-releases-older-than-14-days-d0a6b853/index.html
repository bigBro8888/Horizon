PyPI blocks new file uploads to releases older than 14 days

News time: 2026-07-24T04:29:04.862507+00:00 · Source: rss

Starting July 22, 2026, the Python Package Index (PyPI) has implemented a hard security boundary: no new files — such as wheels or source distributions — can be uploaded to package releases that are more than 14 days old. This change, announced by PyPI maintainer Seth Larson, is not in response to an active attack but rather a proactive defense against a known, technically feasible supply-chain poisoning vector. The policy applies universally across all packages and requires no opt-in — it’s enforced automatically by PyPI’s warehouse backend.

The motivation stems from real-world risks tied to long-lived publishing credentials. Many Python projects use API tokens or GitHub Actions trusted publishers to automate releases. If those credentials are compromised — via leaked secrets, misconfigured CI/CD pipelines, or insider threats — attackers could quietly inject malicious binaries into existing, stable releases. Because older releases are widely depended upon and rarely audited, they make ideal targets for stealthy compromise. As the PyPI team noted, there’s no evidence this technique has been abused yet, but its feasibility was never in doubt.

This isn’t PyPI’s first step toward hardened packaging security — it follows earlier moves like mandatory two-factor authentication for maintainers and the adoption of trusted publishing via OIDC. But unlike those measures, which rely on user adoption or optional configuration, the 14-day upload cutoff is a systemic, immutable guardrail. It eliminates entire classes of post-release tampering without requiring developers to change workflows or understand cryptographic signing.

The impact is subtle but significant. Most well-maintained packages release frequently enough that the restriction won’t interfere with normal operations. However, unmaintained or legacy packages — especially those used in air-gapped or compliance-sensitive environments — may now face challenges if urgent patches require re-uploading binaries. That trade-off is intentional: PyPI prioritizes integrity over convenience, recognizing that trust in the index underpins the entire Python ecosystem. As one security researcher put it, ‘You don’t defend against yesterday’s attacks — you prevent tomorrow’s.’
